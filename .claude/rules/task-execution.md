# 작업 실행 규칙

## 작업 절차

1. `package.json` + 영향 받는 폴더의 기존 패턴 확인 (새로 짓기 전 관례부터 — 재사용 우선순위 `core/` → `ui/` → 신규).
2. 호출 그래프 파악 — 서버 읽기는 서버 컴포넌트가 `network`(`publicFetch` + Next 캐시)를 직접 호출한다. React Query 훅(`clientFetch`)은 검색·낙관적 토글 등 클라 인터랙션 경로에만 낀다.
3. 구현 — 코드는 Learn by Doing으로 사용자가 직접 작성하도록 단계별 유도 (아래 섹션).
4. build/lint 통과 확인 (→ `실행 단위` 참조).
5. 변경한 라우트 수동 스모크 테스트.
6. 진행 중인 Day 작업이면 해당 `docs/plan/roadmap1/week-{N}.md`의 완료 기준 확인.

## 실행 단위

**task 1개 단위로만 진행 후 정지.** 빌드 확인(`pnpm build`·`pnpm lint`)은 해당 task에 포함 — task당 한 번만.

## Learn by Doing

**모든 task에 적용.** Claude는 코드를 직접 작성하지 않는다. 사용자가 순서대로 모든 코드를 작성하도록 유도한다. 단, 오타·한 줄 수정·검증 명령 실행 등 학습 대상이 아닌 작업과 문서·config·rule 등 비(非)구현 작업은 대상이 아니다 (`작업 절차`의 `3. 구현`을 이 섹션이 대체).

**흐름:**

1. **계획 공유** — task를 레이어(API/훅/UI 배선)·설계 결정이 갈리는 경계에서만 쪼갠다 — 보통 2~3개, 자명하면 1개도 무방. 순서와 이유를 먼저 설명한다.
   - **스텝 순서는 "브라우저에서 먼저 눈으로 확인할 수 있는 것"부터** (outside-in). UI 작업이면 붙이는 자리·데이터 fetch를 먼저 잡아 임시 placeholder로 화면에 띄운 뒤, 그 자리를 실제 컴포넌트로 채워 나간다. 하위 컴포넌트·유틸부터 쌓는 bottom-up은 사용자가 전체 그림 없이 따라가게 되므로 지양한다. 컴파일이 안 되는 중간 상태는 placeholder로 메운다.
2. **단계별 요청** — 한 번에 한 단계씩 Learn by Doing 형식(🧭 Context / 🎯 Your Task / 💡 Guidance)으로 요청한다. 사용자 응답 전 다음 단계로 넘어가지 않는다.
   - **🧭 Context는 원래 task가 무엇이었는지 한 줄 환기로 시작**한 뒤, 지금까지 배선된 상태와 이번 단계가 왜 중요한지로 잇는다.
3. **검토 후 다음** — 사용자가 제출한 코드를 검토하고 인사이트를 한 줄 공유한 뒤, 다음 단계를 요청한다.
4. **완료** — 모든 단계가 끝나면 task 완료 보고로 마무리한다.

**제약:** Claude가 `TODO(human)` 플레이스홀더를 포함한 어떤 구현 코드도 작성하지 않는다. 파일 생성·타입 정의·import 추가도 사용자에게 요청한다.

**예외 — `/auto`:** 사용자가 `/auto`를 입력한 task에 한해 이 규칙을 건너뛰고 Claude가 코드를 직접 구현한다 (일회성, 다음 task부터 자동 복귀). 커맨드 정의는 `.claude/commands/auto.md`.

## 2단 ask — 침묵=동의 금지

한 task **안에서** Learn by Doing 4단계가 돌고, task가 끝난 뒤 이 2단 ask를 수행한다. task 종료 후: ① 완료 보고 → 검토 대기 → ② "다음 스텝 시작할까요? — `{요약}`" → 명시적 승인 대기.

## 플랜

플랜은 현재 진행할 task 하나만. 이모지 타이틀 포맷 유지. 상세 작성 규칙:

@rules/plan.md

## task 완료 보고 형식

```
## 📋 Task N — (변경 내용 + 전/후 스니펫)
## 💡 적용한 이유 — 문제 / 왜 이 방식 / 동작 원리
## 🔜 다음 task — Task N+1: {한 줄 요약}
```

## 진행 표기

사용자 확인 후 `week-{N}.md` 해당 task를 `[x]`(상태 열)로 체크. Day의 모든 task가 끝나면 상단 일별 요약 표의 완료 열을 채우고, 그 **요약 행 앞에 ✅ 이모지**를 붙인다.

## 이슈 기록

`week-{N}-issues.md`에 `-` 나열형으로 기록. 본문에는 적지 않는다. **단, 스모크 중 우연히 발견해 즉시 고친 사소한 원라인 버그는 제외** — 설계 결정·재발 가능한 함정·의도적으로 미룬 백로그만 기록 대상.

```
- 핀 클릭이 `<div onClick>` — 키보드 접근 없음 → `<button aria-label>` 로 교체
```
