# 플랜 작성 규칙

현재 task 하나만. 이모지 타이틀. 파일 나열·구현 세부(px·CSS 클래스·함수명) 금지.

## A. 표 구조·표기

- 열 구성: `그룹 | 할 일 | 구현 방식 | 상태`
- **할 일** = 자연어 서술형(무엇을·왜). 함수명·타입명·파일경로 금지 — 사람이 읽는 칸.
- **구현 방식** = "어떻게 붙일지" 짧은 메모.
  - 남길 것: 핵심 엔드포인트, 접근법(서버/클라·fetch 방식·낙관적 토글·미들웨어), 명령어, env.
  - 뺄 것: 함수명·타입명·파일경로(추측성이라 쉽게 어긋남). 정확한 경로·함수명은 구현 시점에 결정.
- 날짜 미정이면 날짜 열을 넣지 않는다.
- 각 Day 헤딩 아래 **목표** — … 한 줄 (요약 표의 목표와 짝).

## B. 태스크 입도 (너무 잘게 쪼개지 말 것)

- 기능 단위 한 행. "타입 정의"와 "API 함수"는 한 행으로 통합(서술 반복 방지).
- 같은 화면의 부수작업(예: mock 삭제)은 그 화면 주작업 행에 흡수 — 독립 행 금지.
- 반복형(에러 페이지 N종, 인증 가드 N단계)은 한 행으로 묶고 세부는 구현 방식에.
- 단, 레이어(API 함수 / 상태관리 훅 / UI 배선)는 분리 유지 — 실행 단위가 다름.
- 완료된 항목은 체크리스트가 아니라 기록이므로 더 공격적으로 병합.
